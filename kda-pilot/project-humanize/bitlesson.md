## Lesson: aiter-fp32yq-mfma-qk
Lesson ID: BL-20260723-aiter-fp32yq-mfma-qk
Scope: testbench/tasks/glm52/dsa_prefill_attn/candidate.py (and any ROCm/MI300X kernel whose matmul needs fp32 output for a correctness gate but currently runs as an fp32 einsum with no MFMA)
Problem Description: The dominant QK score matmul (~60% of the chunked torch sparse-attention kernel) was an fp32 einsum on bf16 inputs upcast to fp32. It ran at ~40 TFLOP/s (no MFMA) because torch has NO bf16-in/fp32-out GEMM: torch.bmm/matmul(bf16,bf16) rounds the output to bf16, and a bf16-rounded QK logit drifts calc_diff to ~6.5e-6, just over the 5e-6 DeepGEMM gate. So the fast MFMA path (bf16 bmm) FAILED correctness and the correct path (fp32 einsum) was slow.
Root Cause: bf16 accumulation of the logits loses precision vs the frozen TileLang reference (which keeps fp32 logits). MI300X MFMA natively does bf16×bf16 with fp32 accumulate, but torch does not expose a bf16-in/fp32-out matmul, so the fp32 accumulator is truncated to bf16 on store.
Solution: Use aiter's tuned `batched_gemm_bf16` (from aiter.ops.triton.gemm.batched.batched_gemm_bf16) for the QK matmul and pass a CALLER-PREALLOCATED fp32 `YQ` output tensor. The kernel writes to YQ's pointer, so the true output dtype is YQ.dtype=fp32; `dtype=torch.bfloat16` is passed ONLY to satisfy the wrapper's `assert dtype in [bf16,fp16]`. bf16×bf16 products are exact in fp32, so the logits are bit-identical to the fp32 einsum apart from fp32 accumulation-order noise (~1e-14 rel). This gives MFMA throughput (~72 TFLOP/s achieved, ~1.6x whole-kernel) at UNCHANGED precision (calc_diff 2.884e-6 == baseline). Guardrails: an import-time probe verifies the dtype=bf16+fp32-YQ path really writes fp32 (rel<1e-4 vs einsum) before enabling; `_qk_scores()` falls back to the fp32 einsum (NEVER a bf16 path) on any exception; no env mutation and no probe inside run() (both at module import, outside timing).
Constraints: The dtype/YQ-dtype mismatch is an aiter contract fragility — if a future aiter build uses `dtype` to cast the store, the probe disables the path and the code falls back to the fp32 einsum (still correct, still ~1.9x vs TileLang). Every authoritative gate run must export AITER_TRITON_ONLY=0 or the sglang TileLang reference silently degrades to the bf16 fallback (see memory glm52-rocm-env-aiter-incomplete). Do NOT accept a bf16-QK candidate that scores calc_diff 0 against the degraded reference — that is a reward-hack against a broken oracle.
Validation Evidence: Authoritative gate (testbench/harness/evaluate_task.py, full sweep M=1024/2048/4096, --repeat 10, cold-L2, conservative q=0.9, AITER_TRITON_ONLY=0), baseline (fp32-QK einsum @ HEAD) vs new, same session: geomean_primary_util_ratio 1.3044→2.1181, min_primary_util_ratio_conservative 1.2603→2.0626, MFU 0.0340→0.0552, whole-kernel M=1024 6519→4009us (~1.62x), calc_diff 2.884e-6 unchanged, shapes_regressed 0, timing_unstable False, post_timing_correct True. Persisted runs/glm52/dsa_prefill_attn/latest.json (run 20260723T025835Z-4fc839). Codex (analyze) green-lit KEEP, no reward-hacking. Round-1 re-gate under CK-restored env held: 3/3 passed, ratio 2.10/2.11/2.13, 0 incorrect.

## Lesson: archive-rebuild-committed-inputs
Lesson ID: BL-20260723-archive-rebuild-committed-inputs
Scope: any COMMITTED rebuild/helper script that reconstructs archived artifacts in this repo (e.g. `archive/0720-Best-GLM-52/lichangye/token_perf/build_token_perf.py`); more generally, any script whose documented "rebuild from the repo" command must work on a fresh checkout.
Problem Description: `codex review` returned a `[P3]`: the archived plot-rebuild helper's `final_result(task, run_id)` read its accepted-result numbers from `runs/glm52/<task>/<run_id>/result.json`. But `runs/` is gitignored (`.gitignore:31`), so on a fresh archive checkout that path does not exist and the README-documented `python build_token_perf.py` rebuild command fails with FileNotFoundError. The script depended on the author's local run cache instead of the archive contents it is supposed to rebuild from — even though the very same `result.json` was committed one directory up.
Root Cause: The rebuild helper and the committed data-of-record live in the SAME repo but at DIFFERENT paths: the authoritative copy is committed at `archive/0720-Best-GLM-52/lichangye/<task>/result.json`, while the script pointed at the transient `runs/glm52/...` gate-output cache (gitignored). A committed script that reads a gitignored sibling is only reproducible on the machine that produced the cache — a self-containment (AC-5) defect, not a numeric one.
Solution: Point the read at the COMMITTED archived copy first and keep the gitignored cache only as a fallback: `archived = OUT.parent / task / "result.json"; cached = REPO / f"runs/glm52/{task}/{run_id}/result.json"; path = archived if archived.exists() else cached`. Report the archived path in the artifact's `source` field. Because the previously path-only `run_id` argument now no longer selects the file, keep it meaningful and preserve provenance by asserting `data["run"]["run_id"] == run_id` (raise on mismatch). Do NOT regenerate committed binary artifacts (plots/CSV) as part of the fix — they may depend on OTHER non-archived inputs (here a hardcoded `~/.claude/...jsonl` transcript loaded at import) and regenerating churns binaries the finding did not ask about; scope the fix to the flagged data source only, and queue the separate non-archived dependency.
Constraints: This is NOT frozen authority (an archive helper, not taskset/metrics/thresholds/references/cost-model/peaks/timing/deploy/candidate/harness/knowledge), so fixing it is in-authority and reviewer-requested — but keep the diff bounded to exactly the one flagged file (`git status --porcelain` must show only it, no `.humanize/`, no CSV/plot churn) and commit with a non-authority scope (`archive:`), no AI-authorship trailer. Verify with `python3 -m py_compile` plus a functional check that replays the new path resolution against the real committed data for every hardcoded (task, run_id) pair (archive path resolves, run_id matches, every consumed field present) before committing. A separate non-archived input the review did NOT flag stays queued, not silently refactored.
Validation Evidence: Round-5 review [P3] on `build_token_perf.py:105`. After the fix: `py_compile` OK; functional replay for all four tasks resolved from the ARCHIVE path with matching run_ids and the accepted-baseline ratios (decode 1.0655, prefill 1.0809, dsa 1.3044, index_score 2.8371), every aggregate field present. `git status --porcelain` = exactly `archive/0720-Best-GLM-52/lichangye/token_perf/build_token_perf.py`; selftest 26/0 and all four knowledge validators (`lint` 17/0, `query`, `index --check` 0 stale, `distill --check` up to date) unaffected. ## Lesson: archive-measured-bytes-provenance
Lesson ID: BL-20260723-archive-measured-bytes-provenance
Scope: any archived accepted-result snapshot in this repo (`archive/**/<task>/result.json` + its sibling `candidate/candidate.py`); any review that checks an archived artifact can PROVE which bytes produced it.
Problem Description: `codex review` returned a `[P2]`: the archived `result.json` records `candidate.sha256`, but the committed archived `candidate/candidate.py` hashed to something else — for ALL FOUR tasks, and the LIVE task candidate ALSO differed from the recorded hash. The archive could not prove that the committed candidate produced the accepted `result.json`.
Root Cause: The accepted `result.json` files were measured at the FIRST-loop candidate commits (the harness records `candidate.sha256 = hashlib.sha256(candidate.py bytes)` via `result_store.sha256_file` at measure time). AFTER measurement, the candidates were edited by later "finalize cleanup" / device-guard / current-loop optimization commits, and the archive's `candidate/candidate.py` snapshot was taken at a still-different (newer) commit (`ebfadea`). So three different byte versions existed (measured / archived-snapshot / live) and only the measured one matches the recorded sha.
Solution: Recover the EXACT measured bytes from git history and place them in the archive — do NOT re-run the gate. For each task, find the commit whose `candidate.py` blob hashes to the recorded sha (`for c in $(git rev-list --all -- <path>); do git cat-file blob $c:<path> | sha256sum; done`), then `git show <commit>:<path> > archive/.../<task>/candidate/candidate.py`. Verify `sha256sum` of each restored file == `result.json` `candidate.sha256`. This makes each archived (result, candidate) pair provably consistent while changing NO numbers, NO live candidate, NO `result.json`, and NO harness/authority.
Constraints: Re-running to "refresh" the hashes is WRONG here: it would change the plan-cited accepted-baseline numbers (e.g. the live dsa candidate scores ~2.11 vs the archived baseline 1.3044) and is not a candidate-local optimization — the archive is a frozen historical baseline the plan references by run_id, so its numbers must not move. Only edit the archive (never the live tree) and never edit `result.json` (that is the authoritative measurement record — falsifying its recorded sha to match current bytes would be dishonest). If the measured bytes are NOT recoverable from history (measured from an uncommitted working tree), you cannot honestly reconcile by byte-recovery — escalate rather than re-run. Keep the diff bounded to the archived candidate files (`git status --porcelain`), commit `archive:` scope, no AI-authorship trailer.
Validation Evidence: Round-6 review [P2]. Recorded shas matched neither archived nor live candidates for all four tasks; git-history search found the measured bytes at first-loop commits (moe_total_decode `7dc4959`→f65b8d4d, moe_total_prefill `3c8aa34`→30d0e47b, dsa_prefill_attn `3531593`→7d8437cd, index_score_prefill `37132ff`→e8438447). After `git show <commit>:… > archive/.../candidate/candidate.py`, all four archived candidates hash-MATCH their `result.json` `candidate.sha256`; `git status --porcelain` = exactly the four archived candidate files; selftest 26/0 and all four knowledge validators unaffected. Committed `b956cc8` (`archive:` scope, no AI-authorship trailer, tree clean).
